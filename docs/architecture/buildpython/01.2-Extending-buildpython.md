# Extending buildpython

This document covers the current extension points for `buildpython`.

## Add a new step

1. Implement a runner that returns `RunResult`.
2. Register the step in `buildpython/steps/step_defs.py` with a stable `number`, `name`, `description`, and log filename under `buildlog/keyrgb/`.
3. If the step depends on an optional tool, keep the skip behavior in `buildpython/core/runner.py` rather than failing at import time.
4. Add the step to the appropriate profiles in `buildpython/core/profiles.py`.

Use stable names and numbers. They are part of the CLI surface for `--run-steps`, `--skip-steps`, and downstream docs.

## Add a debt-oriented step

If a step produces structured debt or architecture data, write stable report files under `buildlog/keyrgb/`.

Current report-oriented steps use predictable `json/csv/md` triplets such as:

- `code-hygiene.{json,csv,md}`
- `architecture-validation.{json,csv,md}`
- `coverage-summary.{json,csv,md}`
- `exception-transparency.{json,csv,md}`

If the new step should appear in the combined outputs, also update:

- `buildpython/core/summary.py` for build-summary debt snapshots and terminal highlights
- `buildpython/core/debt_index.py` for `debt-index.{json,md}`

## Profiles

Profiles are defined in `buildpython/core/profiles.py`.

- Keep `ci` deterministic and reasonably fast.
- Keep `debt` focused on structured debt reporting and ratcheting.
- Keep `quick` useful for frequent local runs.
- Keep `full` as the broader local quality gate.
- Keep `release` aligned with the packaging path used in release automation.

Remember that bare `python -m buildpython` is separate from named profiles. It follows the live registry-minus-Black default in `buildpython/core/cli.py`.

## Typed quality-exception tags

Step-specific inline waivers now use typed quality-exception tags:

```python
# @quality-exception <step-slug>: short reason
```

Example:

```python
# @quality-exception exception-transparency: optional runtime boundary for startup fallback
```

Guidance:

- Use the checker's stable step slug, not a display name.
- Keep the explanation short and local to the guarded code path.
- Treat the slug as step-specific. A waiver for one scanner should not silently waive another.
- If future scanners adopt tags, choose stable step slugs up front and keep them compatible over time.

## Non-flaky step design

- Do not require real hardware in default profiles.
- Do not require network access for normal local or CI checks.
- Prefer repo-local analysis and explicit optional-tool skips over environment-sensitive heuristics.
- Keep outputs plain text or structured data without ANSI formatting inside report files.
