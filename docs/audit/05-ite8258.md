## Summary

The core packet protocol for `0x048d:0xc195` matches public references. KeyRGB correctly
keeps this backend scoped to the **Lenovo Legion 5 Gen10 24-zone keyboard-only path** and
does not fold in the `0xC197` composite keyboard/chassis controller.

No packet-layout bug was found. Two low-risk improvements were made during the audit:

1. Added/sent `SWITCH_PROFILE (0xC8)` reports before writing profile data, brightness, or
   the off profile. This makes writes target the profile KeyRGB saves instead of silently
   depending on whatever profile the firmware currently has active.
2. Added probe identifiers for the OpenRGB-backed HID usage page/usage and feature-report
   size.

The main remaining risk is hidraw interface selection: public references select the device
by VID/PID **and** usage page `0xFF89` / usage `0x07`; KeyRGB's shared hidraw scanner still
filters only by VID/PID.

## Detailed Comparison

### 1. USB ID and device scope

| Source | PID(s) | Interpretation |
|--------|--------|----------------|
| OpenRGB `LenovoDevices.h` | `0xC195` | `LEGION_5GEN10` |
| OpenRGB `RGBController_Lenovo_Gen7_8.cpp` | `0xC195` | `Is24ZoneDevice()` and `IsKeyboardOnlyDevice()` |
| legion-kb-rgb udev rule | `048d:C195` | Lenovo Spectrum keyboard RGB controller |
| KeyRGB `ite8258` | `0xC195` | 24-zone keyboard-only backend |
| KeyRGB `ite8258-chassis` | `0xC197` | Separate composite keyboard/chassis backend |

**Finding:** Correct. `ite8258` should remain `0xC195` only. The existing split with
`ite8258-chassis` matches OpenRGB's `LEGION_5GEN10` vs `LEGION_7GEN10` distinction.

**Action:** Added a comment beside `SUPPORTED_PRODUCT_IDS` documenting this split.

---

### 2. HID usage page / usage

| Source | Usage page | Usage |
|--------|------------|-------|
| OpenRGB detector | `0xFF89` | `0x07` |
| KeyRGB protocol constants | `0xFF89` | `0x07` |

**Finding:** Constants match OpenRGB. Probe output now includes these identifiers.

**Residual risk:** KeyRGB's shared hidraw discovery currently matches only VID/PID. If a
`0xC195` system exposes multiple hidraw interfaces under the same USB ID, KeyRGB may open the
wrong node. OpenRGB avoids this by registering a detector with VID/PID + usage page + usage.

**Recommendation:** Add usage/report-descriptor filtering to the shared hidraw scanner before
promoting this backend beyond experimental. A lightweight first step could confirm report ID
`0x07` with 960-byte feature reports, matching `legion-kb-rgb`'s descriptor check.

### 3. Packet envelope

| Field | KeyRGB | OpenRGB / legion-kb-rgb | Result |
|-------|--------|--------------------------|--------|
| Report ID | `0x07` | `0x07` | ✅ |
| Feature report size | `960` | `960` | ✅ |
| Gen10 length header | bytes 2–3 = payload length | OpenRGB `SetGen10PayloadLength()` | ✅ |
| Save/effect command | `0xCB` | `SAVE_PROFILE` / `EffectChange` | ✅ |
| Brightness command | `0xCE` | `SET_BRIGHTNESS` / `Brightness` | ✅ |
| Switch profile command | `0xC8` | `SWITCH_PROFILE` / `ProfileChange` | ✅ |

**Finding:** Correct. KeyRGB's dynamic little-endian payload-length header is correct for
`0xC195` Gen10, unlike the fixed `0xC0 0x03` framing used by the separately-audited `0xC197`
chassis path.

---

### 4. Profile targeting

OpenRGB reads the current profile ID and writes updates to that active profile. KeyRGB uses a
fixed `profile_id` (default `0x01`) because this backend facade only has a feature-report
writer and no read path.

**Original risk:** KeyRGB wrote profile 1 but did not switch to it first, so writes could be
invisible if the controller was on another profile.

**Resolution:** KeyRGB now sends:

```text
07 C8 01 00 <profile_id>
```

before profile writes, brightness changes, and turn-off writes.

**Trade-off:** This does not preserve the user's previously active firmware profile, unlike
OpenRGB. But it ensures KeyRGB's requested state is visible and matches the approach already
used by `ite8258-chassis`.

---

### 5. 24-zone matrix layout

OpenRGB declares the `0xC195` 24-zone keyboard as:

```text
4 rows × 6 columns
LED IDs: 0x01..0x18
matrix map: 0..23 row-major
```

KeyRGB uses:

```python
NUM_ROWS = 4
NUM_COLS = 6
LED_IDS = tuple(range(0x01, 0x19))
led_id_from_row_col(row, col) = row * 6 + col + 1
```

**Finding:** Correct.

---

### 6. Group payload layout

KeyRGB `_encode_group()` writes:

```text
group_index, 0x06,
0x01 mode,
0x02 speed,
0x03 spin,
0x04 direction,
0x05 color_mode,
0x06 0x00,
color_count, RGB..., led_count, LED IDs little-endian...
```

This matches OpenRGB's `setLedsByGroup()` layout.

**Finding:** Correct. KeyRGB's exact-size packing is at least as safe as OpenRGB's
`PACKET_SIZE - 21` loop guard.

| Concept | KeyRGB | Public reference | Result |
|---------|--------|------------------|--------|
| Screw rainbow | `0x01` | OpenRGB / LLT | ✅ |

**Finding:** Correct. KeyRGB's UI-scale mapping (`0..50` brightness, `0..10` speed) is a
KeyRGB policy layer; public references confirm only the raw hardware ranges.

---

### 8. Capability contract

KeyRGB reports:

```python
BackendCapabilities(per_key=True, color=True, hardware_effects=True, palette=False)
```

This is acceptable under KeyRGB's existing UI contract: the backend exposes a 4×6 editable
matrix of hardware zones, not literal individual physical keys. Keep documenting it as
24-zone hardware so users do not expect per-switch RGB on `0xC195`.
