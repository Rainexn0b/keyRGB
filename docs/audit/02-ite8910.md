**Backend:** `ite8910` — ITE 8910 per-key HID (Clevo/XMG/TUXEDO 2023+ laptops)
**Stability:** VALIDATED
**Priority:** 94
**Source files:**

**External references compared against:**
1. [Chocapikk's blog post](https://chocapikk.com/posts/2026/reverse-engineering-ite8910-keyboard-rgb/) —
   the authoritative reverse-engineering report with complete protocol documentation.
2. [tuxedo-drivers `ite_829x.c`](https://github.com/tuxedocomputers/tuxedo-drivers/blob/master/src/ite_829x/ite_829x.c) —
   the original kernel driver for this chip.
3. [OpenRGB MR !3236](https://gitlab.com/CalcProgrammer1/OpenRGB/-/merge_requests/3236) —
   OpenRGB integration by the same author.

**Note:** The blog post author (Chocapikk) also contributed directly to KeyRGB
via [PR #3](https://github.com/Rainexn0b/keyRGB/pull/3), which added full
direction and color support, and validated all modes on real hardware (XMG PRO
E23). The backend was promoted from EXPERIMENTAL to VALIDATED based on this
validation. This is the strongest possible validation evidence for a backend.

---

## Summary

**No protocol bugs found.** The ite8910 backend faithfully implements the
correct 6-byte HID feature report protocol ([CC, cmd, d0, d1, d2, d3]) as
documented by the RE author. Every command code, LED ID encoding, effect
sequence, and color slot format has been verified against three independent
sources.

## Detailed Comparison

### 1. Report format

| Source | Report format |
|--------|--------------|
| Blog post | `[CC, cmd, d0, d1, d2, d3]` — 6-byte HID feature report |
| tuxedo-drivers | `buf[0]=0xcc, buf[1]=cmd, buf[2]=d0, buf[3]=d1, buf[4]=d2, buf[5]=d3` |
| **KeyRGB** | `_report(*data)` → `bytes((REPORT_ID, *data))` where `REPORT_ID = 0xCC` |
| **Match** | ✅ |

**Historical note:** KeyRGB's original implementation used a 7-byte format
(with a `REPORT_TRAILER = 0x7F` byte). This was identified as incorrect by the
RE and fixed in a previous release. The current code uses the correct 6-byte
format.

### 5. LED ID encoding

| Source | Formula |
|--------|---------|
| Blog | `((row & 0x07) << 5) \| (col & 0x1F)` — row 0 = top (F-keys) |
| tuxedo-drivers | `get_led_id(row, col) = (u8)(((row & 0x07) << 5) \| (col & 0x1f))` |
| **KeyRGB** | `((hardware_row & 0x07) << 5) \| (int(col) & 0x1F)` where `hardware_row = (NUM_ROWS - 1) - logical_row` |
| **Match** | ✅ (with row-flip for KeyRGB's bottom-up convention) |

KeyRGB uses a bottom-up logical row convention (row 0 = bottom/modifiers,
row 5 = top/F-keys). The `led_id_from_row_col()` function flips the row
before encoding:

```
logical_row 0 (bottom) → hardware_row 5 → LED IDs 160-179
logical_row 5 (top)    → hardware_row 0 → LED IDs 0-19
```

This mapping was contributed and hardware-validated by the protocol RE author.
The `KNOWN_LED_IDS` tuple (used for bulk operations like `set_color()` and
`reset()`) generates unflipped hardware-order IDs which is correct since those
operations don't depend on logical row ordering.

### 6. Brightness/speed command

| Source | Format | Notes |
|--------|--------|-------|
| Blog | `[CC, 09, brightness, speed]` | brightness 0-10, speed 0-10 |
| tuxedo-drivers | `keyb_send_data(dev, 0x09, brightness, 0x02, 0x00, 0x00)` | **Hardcodes speed = 0x02** (bug in kernel module) |
| **KeyRGB** | `build_brightness_speed_report(b,s)` → `[CC, 09, b, s, 0, 0]` | Uses caller-provided speed ✅ |
| **Match** | ✅ (KeyRGB matches the blog, which is authoritative) |

**Notable finding:** The tuxedo-drivers kernel module hardcodes `speed=0x02`.
KeyRGB correctly passes the caller's speed value through, matching the
documented protocol. KeyRGB's behavior is correct; the kernel module's is
limited.

### 10. Effect parameter order (animation mode vs. color command)

**KeyRGB** sends animation mode first, then color/slot commands. The blog
describes the same pattern (animation mode `[CC, 00, mode]` is a separate
command from color `[CC, 0A, AA, R, G, B]`). The tuxedo-drivers kernel module
confirms this separation by sending animation and color as separate
`keyb_send_data` calls.

**Finding:** Ordering is correct. ✅

---

### 11. Speed policy

**KeyRGB** sets:
```python
keyrgb_hw_speed_policy = "direct"
```

This means UI speed values are passed directly to the firmware without
inversion (unlike the ite8291r3 backend which uses "inverted" because higher
speed values = slower animation on that controller). The blog confirms the
8910 uses speed 0-10 where 0 = slowest, 10 = fastest. This is "direct"
ordering. ✅

---

### 12. Per-key mode policy

**KeyRGB** sets:
```python
keyrgb_per_key_mode_policy = "init_once"
```

This means per-key colors are set once and persist until changed, rather than
requiring re-send on every frame. The blog confirms that per-key writes via
`[CC, 01, led_id, R, G, B]` are persistent. ✅

### 13. Brightness range mapping

| Source | Raw range | UI mapping |
|--------|-----------|------------|
| Blog | 0-10 (4-level UI maps to {2, 4, 6, 10}) | Non-linear Windows behavior |
| tuxedo-drivers | 0-10 | Direct passthrough |
| **KeyRGB** | 0-10 | Linear `ceil(ui * 10 / 50)`, round-trip via `round(raw / 10 * 50)` |

KeyRGB uses a linear mapping from UI 0-50 to raw 0-10, which is cleaner than
the Windows app's hardcoded {2, 4, 6, 10}. The round-trip via
`ui_brightness_from_raw()` is consistent.

**Finding:** Correct, and arguably better than the Windows reference. ✅

### 14. `is_off()` / `get_brightness()` — software tracked vs. hardware query

Unlike the ite8291r3 backend which queries the hardware for state, the ite8910
backend tracks brightness in software via `_current_brightness`:

```python
def turn_off(self) -> None:
    self.set_brightness(0)

def is_off(self) -> bool:
    return self._current_brightness <= 0

def get_brightness(self) -> int:
    return int(self._current_brightness)
```

The blog mentions `GetLedData` exists in the Windows DLL but doesn't document
its response format. Software tracking is the only practical approach for this
controller.

**Finding:** Correct by necessity. No hardware read-back command is publicly
documented for this controller. ✅

### 17. Error handling in `_send()`

```python
def _send(self, report: bytes) -> None:
    result = self._send_feature_report(bytes(report))
    if result == -1:
        raise OSError("Could not send ITE 8910 feature report")
```

**Finding:** The error check is against `== -1`, but the `HidrawFeatureTransport.send_feature_report()`
returns `len(payload)` on success (which is always positive for a valid
report). The ioctl returns a non-negative value on success, and -1 on failure
with errno. This is correct for the Linux ioctl convention. ✅

## Issues Found

### 🔴 Critical: None

### 🟡 Medium: None

### 🟢 Low: Observations

1. **`reset()` sends redundant per-key black after ClearColor**
   - File: `device.py` lines 133-136
   - `reset()` sends ClearColor (`[CC, 00, 0C]`) which already resets all
     LEDs, then sends `(0,0,0)` to all 120 LED IDs individually. The blog says
     ClearColor alone is sufficient. The per-key black writes are redundant
     but harmless (the LED IDs are correct).
   - **Impact:** 120 extra HID reports per reset. Negligible.

2. **No `turn_off()` restores last brightness**
   - `turn_off()` sets brightness to 0. `set_brightness()` simply applies the
     value. There's no "last brightness before off" tracking in the device
     layer (this is handled at the tray/controller level).
   - **Impact:** By design, consistent with ite8291r3 pattern. No issue.

3. **Brightness mapping uses `ceil()` for UI→raw, `round()` for raw→UI**
   - `raw_brightness_from_ui`: `ceil(ui * 10 / 50)` — maps UI 1 to raw 1
   - `ui_brightness_from_raw`: `round(raw / 10 * 50)` — maps raw 1 to UI 5
   - This creates a minor asymmetry: setting UI brightness 1 → raw 1, reading
     back gives UI 5. The ite8291r3 backend avoids this by tracking the
     original UI value, but this backend's `set_brightness()` saves the
     requested UI value directly (line 131: `self._current_brightness = requested`),
     so the round-trip asymmetry only affects code paths that read from
     `ui_brightness_from_raw()` (e.g., after a `set_brightness_raw()` call).
   - **Impact:** Minimal. Only affects internal state tracking if raw
     brightness is set directly.

## Comparison with tuxedo-drivers kernel module

The `ite_829x.c` kernel module serves as a secondary reference. It confirms:

- The 6-byte report format with report ID 0xCC ✅
- The LED ID encoding formula ✅  
- The `[CC, 01, led_id, R, G, B]` per-key command ✅
- The `[CC, 09, brightness, speed]` brightness/speed command ✅
- The `[CC, 00, 09]` random animation command ✅

However, the kernel module is limited compared to KeyRGB:
- It hardcodes `speed=0x02` (KeyRGB passes caller speed correctly)
- It only exposes one animation mode (Random) via sysfs
- It doesn't support ClearColor before per-key updates
- It doesn't support the color-effect commands (breathing, flashing, wave
  direction, etc.)

These limitations are in the kernel module, not in KeyRGB. KeyRGB's
implementation is more complete. ✅

## Conclusion

The ite8910 backend is **correct and complete**. It benefits from being
directly authored/reviewed by the person who reverse-engineered the protocol
(Chocapikk) and validated on real hardware (XMG PRO E23). The protocol
implementation matches all three independent references byte-for-byte.

The only observations are minor (redundant per-key black after ClearColor) and
have no functional impact.

**Recommendation:** No changes needed. This is a well-validated backend.
