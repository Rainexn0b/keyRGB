# 0.30.x Maintainability Sprint

## Purpose

The `0.30.x` series is a **maintainability sprint**: every patch in the series is
expected to be refactor-only, behavior-parity, and gated by the full
`buildpython` release profile. No new hardware features, no new backends, no
user-facing flag changes.

This doc is the **sprint umbrella**. Each planned workstream has its own section
here and a progress log at the bottom. Per-slice detail lives in the section,
not in a separate plan file, unless a workstream grows large enough to deserve
one.

## Relationship to prior plans

- **Does not supersede** `docs/I-implementation-plans/2026-07-15/maintainability-debt-paydown-plan.md`.
  That doc is still the authoritative inventory (D1–D15). Sprint workstreams
  reference inventory IDs inline.
- **Builds on** `docs/I-implementation-plans/2026-07-16/0.30.1-maintainability-follow-up-plan.md`
  (W1/W2/W3 + CQ1–CQ5). The 0.30.1 follow-up is **done**; this sprint takes the
  residual W2 backlog as the entry point, then expands to the next workstream.
- Per-slice progress notes are also reflected back into the D-inventory progress

## WS1 — File-size hotspots

### Goal and entry state

Reduce the >350 LOC production backlog by extracting pure helpers using the
already-proven `_decisions.py` / `_render_brightness_*` pattern. The goal is not
"hit a number" — it is to reduce the *where do I edit?* friction that the
current hotspots cause, while keeping the regression gate (`buildpython
--run-steps=6,12`) honest.

**Entry snapshot (post-0.30.1, `buildpython --run-steps=6`):**

- Refactor (350–399 LOC): **13** prod files
- Critical (400–499 LOC): **10** prod files
- Severe (500–599 LOC): **0** prod files
- Extreme (600+ LOC): **1** prod file
- **Total >350 LOC: 24** prod files

Files excluded by the file-size analyser's `src/` + `buildpython/` scope (test
megasuites) are tracked under **WS3**, not here.

### Extraction pattern (preferred lever)

The proven pattern, established in 0.30.0/0.30.1:

1. Identify a hotspot that mixes **pure decision/plan functions** with I/O,
   side effects, or tray-state mutation.
2. Move the pure functions into a sibling module under the same package, named
   with a leading underscore if internal (`_foo_helpers.py`, `_decisions.py`,
   `_runtime.py`, etc.).
3. The original module re-imports the moved helpers so monkeypatch seams and
   public import paths stay stable.
4. Unit tests for the pure helpers do **not** need the full tray fake surface.
5. Result: the hotspot's LOC drops one band, and the next behavior change is
   "classify → plan → execute" rather than inline branching.

This pattern only applies where the hotspot is genuinely mixed. Files that are
large because they contain one cohesive unit (a single class, a single Tk
window builder, a device-local packet table) are **accepted with a written
waiver** instead of force-split.

## Process

### Per-slice workflow

1. Pick one row from the active workstream.
2. Open a focused branch.
3. Apply the extraction pattern (or the workstream-specific lever).
4. Run the per-slice verification commands above.
5. Update this doc's progress log with: file(s) touched, LOC delta, parity
   tests run, status.
6. PR.

### Sprint cadence

- Each patch in `0.30.x` is **one or more slices** from this doc.
- There is no fixed number of patches; the sprint ends when WS1 + at least one
  of WS2/WS3/WS4 are done, or when the maintainability bar (composite score in
  the maintainability review) reaches the agreed threshold.
- A patch may ship with **zero slices landed** if it is a release-hygiene bump
  only (e.g. dependency pin refresh) — but that should be the exception.

### Roll-forward policy

A slice that does not land in the patch it was aimed at is **not cancelled**.
It moves to the next patch's plan with a one-line note here. Cancelling a slice
requires either (a) the underlying assumption changed, or (b) the slice proved
larger than the sprint and is promoted to its own plan file under this date
folder.
