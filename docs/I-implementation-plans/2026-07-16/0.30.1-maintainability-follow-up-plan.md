# 0.30.1 Maintainability Follow-up Plan

## Purpose

Post-`0.30.0` maintainability patch. The 2026-07-15 debt wave (D1–D6 primary
slices) is largely **monitoring / done**. This plan targets the **three residual
weaknesses with the best ROI for a short release**, plus bounded code-quality
fixes that keep the quality gates honest without a packaging rewrite or product
feature work.

Supersedes near-term ordering only; inventory IDs still live in
`docs/I-implementation-plans/2026-07-15/maintainability-debt-paydown-plan.md`.

## Three biggest weaknesses (now)

### W1 — Orchestration density still owns behavior (D2 / D5 residual)

**Problem:** Pure gates landed for config-apply mode and power-source skip, but
pollers/controllers still own large side-effect graphs. Changing one power or
secondary-device behavior still touches multiple modules and fat fakes.

**Evidence:**

- `src/tray/pollers/hardware_polling.py` (~478 LOC)
- `src/tray/idle_power_state.py` (~497 LOC)
- `src/tray/pollers/idle_power/_runtime.py` (~428 LOC)
- `src/core/power/management/manager.py` (~434 LOC)
- `src/tray/controllers/lighting_controller.py` + helpers (~356 + ~414 LOC)

**Why now:** Highest contributor friction after 0.30.0; pure-plan expansion is
already the proven pattern.

**Exit criteria:**

- At least two more pure decision functions extracted (no I/O) from
  hardware-poll or idle-power apply path, with unit tests that do not need a
  full tray fake surface.
- Poller/controller methods become thin: classify → plan → execute.
- No intentional behavior change (parity tests green).

### W2 — Production size hotspots concentrate risk (D14 residual)

**Problem:** Almost all prod files are well-split; a few remain in
refactor/severe buckets and mix protocol, coordination, and rendering policy.

**Evidence (largest prod):**

| LOC | Path | Split direction |
|---:|---|---|
| ~709 | `src/core/backends/ite8258_perkey_chassis/protocol.py` | Keep packets local; extract pure profile/frame helpers if still mixed |
| ~546 | `src/core/effects/reactive/_render_brightness_support.py` | Split pure brightness math vs render side effects |
| ~497 | `src/tray/idle_power_state.py` | Move pure predicates/helpers out of state bag if still mixed |
| ~478 | `src/tray/pollers/hardware_polling.py` | Align with W1 pure plan extraction |

**Why now:** These are the only remaining “where do I edit?” magnets; splitting
them locks in the 0.30.0 file-size culture.

**Exit criteria:**

- No new prod file above the severe LOC threshold without a documented waiver.
- Target: move at least **one** severe file under the critical band via pure
  extract (prefer reactive brightness support or hardware polling).
- Chassis protocol may stay large if truly device-local packet tables; only
  extract pure non-I/O helpers when obvious.

### W3 — Confidence gap on multi-layer runtime flows (D7 residual)

**Problem:** Unit coverage is excellent (~86%); issue-class bugs (e.g. #7
composite + forced-off + config apply) still need multi-layer paths. One
integration landed in 0.30.0; more cross-layer coverage prevents silent
regressions when W1/W2 refactors land.

**Evidence:**

- Strong units under `tests/tray/pollers/**`, `tests/core/**`
- Existing: `tests/tray/test_composite_config_apply_forced_off_integration_unit.py`
- Thinner: config-apply → secondary static → power restore as one flow
- GUI package coverage lowest; mypy does not typecheck `src/gui`

**Why now:** Refactors without multi-layer tests re-create issue #7 risk.

**Exit criteria:**

- Add **2** hardware-free multi-layer tests:
  1. Config apply (uniform/effect) + secondary static scene + forced-off skip/resume
  2. Idle/power dim → restore path that asserts brightness bag + lighting
     controller call order via owner-backed fakes (no real USB)
- Optional small quality add-on (if time): enable mypy on a **narrow** GUI subset
  (`src/gui/settings` or pure modules only) with a baseline, not full Tk surface.

## Release constraints

- Version: `0.30.1` (patch; no user-facing feature flags required).
- Behavior: **parity-first**. Refactors only; fix bugs only if found by new tests.
- Gates: full `buildpython` profile used for release (pytest, ruff, architecture,
  exception transparency, loc-check, coverage). Do not expand exception budgets.
- Tests: keep `KEYRGB_DISABLE_USB_SCAN=1` defaults; no hardware required for CI.
- Docs: update this plan’s progress log; link from
  `docs/I-implementation-plans/README.md` and progress on 2026-07-15 inventory
  (D2/D5/D7/D13/D14 status notes).
- Changelog: under `## 0.30.1` maintainability bullets only.
