# KeyRGB Optimisation Notes

This folder tracks concrete optimisation findings and the next profiling slices.

## Memory baseline: 2026-03-31

Goal:

- establish a real idle RAM baseline for the running tray process before changing code
- distinguish a fixed startup footprint from leak-like growth

Sampling notes:

- a second `./keyrgb` instance cannot be used for measurement because KeyRGB is single-instance and exits after detecting the existing lock
- the useful target was the already-running tray process:
  - command: `/home/cyril/Projects/Keyrgb/.venv/bin/python -B -m src.tray`
  - pid during sampling: `1043741`
- this machine did not have an external `time` binary, so the measurement used:
  - `/proc/<pid>/status`
  - `/proc/<pid>/smaps_rollup`
  - `ps`
- two samples were taken 5 seconds apart while the tray was idle

Observed values:

- `VmRSS = 110736 kB` which is about `108.1 MiB`
- `VmHWM = 110736 kB` which matched current RSS during sampling
- `Pss = 82707 kB` which is about `80.8 MiB`
- `Private_Dirty = 63700 kB` which is about `62.2 MiB`
- `Private_Clean = 12264 kB` which is about `12.0 MiB`pyproject.toml
- `Shared_Clean = 34772 kB` which is about `34.0 MiB`
- `VmSize = 1434356 kB`
- `Threads = 16`

Stability result:

- the second sample was identical to the first sample
- this does not currently look like an idle leak or steady-state growth problem
- the more likely target is fixed resident footprint from startup imports, GUI/tray/native integrations, eagerly created runtime services, and large persistent Python objects

## What the numbers likely mean

The process is a mixed Python and native desktop app workload, so RSS alone is not enough.

Important interpretation points:

- `Shared_Clean` is fairly large, which is consistent with shared libraries, Python runtime pages, and GUI/integration libraries
- `Private_Dirty` is the most actionable number because it represents process-private resident pages that KeyRGB is actively keeping alive
- `Pss` is the best rough "real cost" number for comparing optimisations across runs because it discounts shared pages
- the flat second sample means optimisation work should focus on reducing baseline allocations, not chasing a leak first

## Startup/logging observations

Useful runtime observations from the debug session:

- backend probing ran for `asusctl-aura`, `ite8291r3`, `ite8910`, `ite8297`, and `sysfs-leds`
- startup diagnostics emitted a large hardware/config snapshot before the tray reached steady state
- the app uses the AppIndicator backend via `pystray`
- the live process had the USB device open while idle
- the tray process reached a steady idle state successfully on this run

These observations matter because they identify likely fixed-cost contributors before deeper tooling is added.

## Priority optimisation strategy

### 1. Split Python heap from native/library allocations

Do this first before changing behavior.

Recommended tools:

- `tracemalloc` for Python object growth and import-time heap attribution
- `memray` if available for fuller allocator visibility
- `/proc/<pid>/smaps_rollup` before and after feature gates to confirm resident impact

Reason:

- if most of the footprint is native GUI/tray/libdbus/libappindicator state, pure Python micro-optimisations will not move RSS much
- if a large fraction is Python heap, import/lazy-init/data-shape changes will be worthwhile quickly

### 2. Measure startup imports explicitly

Target the tray startup path:

- `src.tray`
- tray icon/integration bootstrap
- backend registry and startup diagnostics
- config/profile loading

Recommended command direction:

- Python import profiling with `-X importtime`

Reason:

- for a desktop tray app, a surprising amount of resident memory can come from modules imported unconditionally during startup even when their functionality is not used immediately

### 3. Defer non-essential startup work

Good candidates to audit for lazy init:

- startup diagnostics snapshot generation
- tray icon image decode and image helpers
- theme detection and UI asset preparation
- power monitoring setup when the user has not enabled related features
- optional backend probing that does not affect the chosen backend on this hardware

Reason:

- the current numbers are stable, which suggests baseline work is happening eagerly and then staying resident

### 4. Reduce duplicated in-memory state in tray/runtime services

Focus on long-lived tray objects and caches:

- config snapshots and repeated tuple/dict conversions
- cached icon/image objects
- backend probe result structures retained longer than necessary
- per-key colour maps or profile structures loaded even when the active effect does not need them

Reason:

- `Private_Dirty` is the main actionable bucket, and long-lived Python containers are one of the easiest ways to inflate it

### 5. Audit always-on background services

Candidates:

- power management manager
- config polling internals
- icon polling / hardware polling / idle-power polling
- any watchers or helper subprocesses started before they are needed

Reason:

- even if CPU usage is fine, background services can pin data structures, threads, and native library state
- the running process had `16` threads, which is not necessarily bad, but it is worth understanding which services own them

### 6. Keep optimisation experiments measurement-driven

For each change, re-check at least:

- `VmRSS`
- `Pss`
- `Private_Dirty`
- thread count
- startup time if lazy loading is introduced

Reason:

- some changes reduce Python heap but do not reduce resident memory
- some lazy-init changes reduce startup RSS but regress first-use latency; both should be recorded

## Suggested next profiling slices

Ordered by expected signal quality:

1. Add a dedicated tray memory profiling mode that records `tracemalloc` snapshots shortly after startup and after 10-30 seconds idle.
2. Add an import-time profiling note for the `src.tray` entrypoint and identify the heaviest unconditional imports.
3. Gate startup diagnostics behind a lazy or explicit debug-only path and re-measure RSS/PSS.
4. Audit tray icon and image-loading paths for duplicated PIL/PhotoImage lifetimes.
5. Audit backend probing for work that can be skipped once the active hardware/backend is already known on this machine.

## Current conclusion

As of 2026-03-31, KeyRGB's tray process on this machine is roughly:

- `108 MiB` RSS
- `81 MiB` PSS
- `62 MiB` private dirty resident memory

The baseline was flat across repeated idle samples. The next optimisation pass should treat this as a fixed-footprint reduction problem, not a leak investigation.
