## Summary

The core per-key row protocol is correct for hidraw/hidapi use. KeyRGB matches
OpenRGB and tuxedo-drivers on:

- 8-byte feature/control reports
- per-key mode `0x33`
- turn-off report `08 01 00 00 00 00 00 00`
- 6 rows × 21 columns
- row announce report `16 00 <row> 00 00 00 00 00`
- 65-byte row output reports with two leading zero bytes followed by B/G/R planes

No protocol-breaking bug was found. One small runtime issue was fixed: zero-brightness
`set_color()` / `set_key_colors()` now stops after the off report instead of sending row
data after power-off.

The main unresolved audit risk is **USB PID coverage**. Public references do not agree with
KeyRGB's current experimental PID list, and no public reference found during this audit
supports `0x6008`.

## Detailed Comparison

### 1. USB ID coverage

| Source | Product IDs | Notes |
|--------|-------------|-------|
| **KeyRGB `ite8291`** | `0x6004`, `0x6008`, `0x600B`, `0xCE00` | Experimental hidraw backend |
| pobrn `ite8291r3-ctl` | `0x6004`, `0x6006`, `0x600B`, `0xCE00` | Requires `bcdDevice == 0x0003` |
| OpenRGB ClevoKeyboardController | `0x600B` | Hidapi detector only |
| tuxedo-drivers `ite_8291.c` | `0x6004`, `0x600A`, `0x600B`, `0xCE00` | Kernel HID driver; routes `CE00 bcd=0x0002` to 4-zone mode |
| Local KeyRGB genAI notes | `0x6008` mentioned | Not independent hardware evidence |

**Finding:** `0x6008` is carried by KeyRGB's experimental backend but was not found in
OpenRGB, pobrn, or tuxedo-drivers. Conversely, references mention `0x6006` and `0x600A`,
which are not in this backend's PID tuple.

**Impact:** Low immediate risk because the backend is experimental and disabled unless the
user opts in. But the table should not be treated as validated hardware coverage.

**Recommendation:** Do not add or remove ITE USB IDs blindly. Gather diagnostics for real
`048d:6008`, `048d:6006`, and `048d:600A` devices before changing the list. If diagnostics
confirm matching report descriptors and successful row writes, add tests and update the PID
table in a dedicated hardware-evidence change.

**Code change:** Added source comments beside `SUPPORTED_PRODUCT_IDS` documenting this split
so future edits do not mistake the current tuple for fully reference-backed coverage.

### 2. CE00 zone-only firmware split

| Source | Behavior |
|--------|----------|
| tuxedo-drivers | `0xCE00` + `bcdDevice == 0x0002` routes to 4-zone writer |
| KeyRGB `ite8291` | Rejects `0xCE00` + `bcdDevice == 0x0002` as zone-only |
| KeyRGB `ite8291_zones` | Separate backend owns the zone protocol |

**Finding:** Correct. The native per-key backend rejects the firmware variant that tuxedo-
drivers routes to zone mode.

**Risk:** Low. Unknown `CE00` bcdDevice values other than `0x0002` are still allowed by this
experimental backend, matching tuxedo-drivers' "everything else is per-key" behavior.

### 4. Row data report layout

KeyRGB uses:

```text
65 bytes total
byte 0..1:  zero padding / report-id convention
byte 2..22: B0..B20
byte 23..43: G0..G20
byte 44..64: R0..R20
```

This matches OpenRGB and tuxedo-drivers exactly. pobrn's PyUSB reference uses B/G/R offsets
starting at byte 1, not byte 2. That appears to be a transport convention difference: KeyRGB
uses hidraw output reports, where the first byte is the HID report ID, matching hidapi and
the kernel driver's `output_report` path.

**Finding:** Correct for KeyRGB's hidraw transport.

**Code change:** Added a docstring to `build_row_data_report()` documenting why KeyRGB uses
offsets 2/23/44 even though the older PyUSB reference starts at offset 1.

---

### 5. DMI/model color scaling quirks

tuxedo-drivers contains a `color_scaling()` quirk table for several TUXEDO/Schenker chassis.
It adjusts red/green/blue differently for models such as Stellaris/Polaris variants and
some `0x600A` / `0x600B` devices.

KeyRGB does not apply DMI-specific scaling in this backend.

**Impact:** Not a protocol bug, but colors may appear too pink/green/blue on affected
hardware compared with tuxedo-drivers.

**Recommendation:** Prefer KeyRGB's existing calibration/profile workflow over hardcoding
DMI quirks in this experimental backend. If users provide diagnostics and measured color
evidence, consider adding model-scoped calibration defaults rather than protocol-level
channel scaling.

---

### 6. Zero-brightness row writes

**Original behavior:** `set_color(..., brightness=0)` and `set_key_colors(..., brightness=0)`
called `turn_off()` through `enable_user_mode()`, then still wrote all row data.

**Risk:** Usually harmless, but unnecessary. It could also turn an already successful off
operation into a user-visible error if a later row write failed.

**Fix:** Both methods now return immediately when `enable_user_mode()` leaves the device off.

**Tests added:**
- `test_device_set_color_zero_brightness_turns_off_without_row_writes`
- `test_device_set_key_colors_zero_brightness_turns_off_without_row_writes`

---

### 7. Capabilities and public contract

KeyRGB reports:

```python
BackendCapabilities(per_key=True, color=True, hardware_effects=False, palette=False)
```

This backend only implements direct per-key color mode. Although the controller supports
hardware effects and palette commands in the broader protocol, those are already covered by
the validated `ite8291r3` backend; keeping this experimental hidraw backend direct-only is a
reasonable public contract.
