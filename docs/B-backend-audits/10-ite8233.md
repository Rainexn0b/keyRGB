## Summary

**No protocol bugs for the OpenRGB-corroborated path (PID 0x7001).** The 8-byte
report format, command bytes, mode IDs, brightness range, speed inversion, and
turn-off sequence for `0x7001` are byte-for-byte matches with OpenRGB's
`ClevoLightbarController`.

However, KeyRGB supports two additional PIDs (`0x7000`, `0x6010`) and a number
of per-PID quirks that are **not corroborated by OpenRGB**. There are also
metadata and permission gaps that affect real-world usability:

1. **Missing udev rules** for all three PIDs — users will hit permission denied.
2. **Usage page mismatch** in probe identifiers (`0xFF89` documented vs `0xFF03`
   used by OpenRGB for `0x7001`).
3. **Naming ambiguity** — OpenRGB identifies the chip as "ITE 8291 rev 0.03",
   while KeyRGB calls it "ITE 8233".
4. **Scan mode not implemented** — OpenRGB exposes it; KeyRGB defines the
   constant but does not handle it.

### 2. Packet layout (PID 0x7001)

**KeyRGB `build_color_packet()` for 0x7001:**
```
[0]  0x14       command
[1]  0x00       variant
[2]  0x01       slot
[3]  red
[4]  green
[5]  blue
[6]  0x00
[7]  0x00
```

**OpenRGB `SetColor()`:** identical (`0x14 0x00 0x01 R G B 0x00 0x00`).

**KeyRGB `build_mode_packet()` for 0x7001:**
```
[0]  0x08       command
[1]  0x22       mode variant
[2]  mode
[3]  speed
[4]  brightness
[5]  apply byte
[6]  0x00 / direction
[7]  0x00
```

**OpenRGB `SetMode()`:** `0x08 0x22 MODE inverted_speed BRIGHTNESS 0x01 0x00 0x00`.

**Finding:** ✅ Byte-for-byte match.

**Finding:** Only `0x7001` is independently corroborated. The other two PIDs and

### 5. Usage page / usage metadata mismatch

| Source | Usage Page | Usage |
|--------|------------|-------|
| KeyRGB probe identifiers | `0xFF89` | not recorded |
| OpenRGB detector | `0xFF03` | `0x02` |

**Finding:** ⚠️ The usage page documented in KeyRGB (`0xFF89`) does **not** match
OpenRGB's detector for `0x7001` (`0xFF03`).

**Impact:** Low functional impact — KeyRGB's hidraw matching does **not** filter
on usage page; it matches by VID/PID only. The value only appears in probe
identifiers and could mislead diagnostics.

**Possible explanation:** `0xFF89` may be correct for `0x7000` or `0x6010`, while
`0xFF03` is correct for `0x7001`. Or KeyRGB's value is simply stale/wrong.

**Recommendation:** Verify per-PID descriptors with `lsusb -v` / `hid-decode` on
real hardware. Until then, document the discrepancy.

---

### 6. Feature report probe mismatch

KeyRGB defines:
- `FEATURE_REPORT_ID = 0x5A`
- `FEATURE_REPORT_SIZE = 16`
- `build_feature_probe_report()` creates a 16-byte packet starting with `0x5A`

OpenRGB sends 8-byte feature reports with **no report ID**; the first byte is
the command (`0x14`, `0x08`, `0x12`, `0x1A`).

**Finding:** ⚠️ The 16-byte `0x5A` probe report is not used anywhere in the
active code path. It only appears in probe identifiers. It may describe an HID
descriptor feature report ID, but it does not match the runtime packets.

**Recommendation:** Remove or correct probe identifiers once descriptor evidence
is available.

---

### 7. Naming note

OpenRGB header: `Driver for Clevo laptop lightbar (ITE 8291 rev 0.03)`  
KeyRGB backend name: `ite8233`

**Finding:** ⚠️ The chip appears to be **ITE 8291 rev 0.03**, not ITE 8233.
OpenRGB's naming is consistent with the canonical `ite8291r3` backend in this
repo.

**Impact:** Naming-only. Under the agreed convention (`ITE<chip>_<capability>`),
this backend would ideally be `ite8291_lightbar` or `ite8291_lb`.

**Recommendation:** Do not rename now (breaking change). Document the ambiguity
and consider an alias in a future cleanup pass.

---

### 8. Scan mode gap

OpenRGB exposes `Scan` mode (`MODE_SCAN = 0x06`) with `MODE_FLAG_HAS_PER_LED_COLOR`,
`MODE_FLAG_HAS_BRIGHTNESS`, and `MODE_FLAG_HAS_SPEED`.

KeyRGB:
- Defines `MODE_SCAN = 0x06`
- Does **not** handle `"scan"` in `Ite8233LightbarDevice.set_effect()`
- Does not advertise any effects in `Ite8233Backend.effects()`

**Finding:** ⏳ Scan is a known gap. The protocol shape for `0x7001` is derivable
from OpenRGB (`0x08 0x22 0x06 inverted_speed brightness 0x01 0x00 0x00`, with
color set first), but the 7-slot color behavior for KeyRGB's multi-slot protocol
is not confirmed.

**Recommendation:** Leave as follow-up. Add scan support only after confirming
color-slot behavior on real hardware or in tuxedo-drivers.

---

### 9. Capabilities

```python
BackendCapabilities(per_key=False, color=True, hardware_effects=False, palette=False)
```

This is correct for a single-zone lightbar with direct color control. The hidden
hardware effects in `device.py` are not advertised, which is appropriate for an
experimental backend.

---

### 10. Missing udev rules (functional bug)

**Finding:** 🐛 `system/udev/99-ite8291-wootbook.rules` does not contain any
entries for `0x6010`, `0x7000`, or `0x7001`. Users with these devices will get
permission denied when KeyRGB tries to open the hidraw node.

**Fix:** Add hidraw uaccess rules for all three PIDs.
