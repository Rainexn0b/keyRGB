## References

1. **tuxedo-drivers** `src/ite_8297/ite_8297.c` — the canonical kernel driver for the ITE 8297
   RGB LED controller. This is the sole public protocol reference; OpenRGB does not cover this
   device.

## Summary

**No protocol bugs found.** The KeyRGB `ite8297` backend is a byte-for-byte match to the
tuxedo-drivers kernel driver. Every aspect of the HID feature report — size, report ID,
command bytes, RGB field order, and USB VID/PID — is correct.

The backend is intentionally conservative: it exposes uniform color writes only, no per-key
control, and no hardware effects. This matches the TUXEDO driver's capability surface.

No code changes were required.

### 2. HID feature report format

**tuxedo-drivers `ite8297_write_color()`:**
```c
#define HID_DATA_SIZE 64

buf[0] = 0xcc;
buf[1] = 0xb0;
buf[2] = 0x01;
buf[3] = 0x01;
buf[4] = red;
buf[5] = green;
buf[6] = blue;

hid_hw_raw_request(hdev, buf[0], buf, HID_DATA_SIZE,
                   HID_FEATURE_REPORT, HID_REQ_SET_REPORT);
```

**KeyRGB `build_uniform_color_report()`:**
```python
PACKET_SIZE = 64
REPORT_ID = 0xCC
COMMAND_COLOR = 0xB0
COMMAND_DIRECT = 0x01
COMMAND_SUBDIRECT = 0x01

packet = bytearray(PACKET_SIZE)
packet[0] = REPORT_ID          # 0xCC
packet[1] = COMMAND_COLOR      # 0xB0
packet[2] = COMMAND_DIRECT     # 0x01
packet[3] = COMMAND_SUBDIRECT  # 0x01
packet[4] = red
packet[5] = green
packet[6] = blue
```

**Finding:** ✅ Byte-for-byte match. Both use a 64-byte feature report with report ID `0xCC`.

---

### 3. Transport path

| Layer | tuxedo-drivers | KeyRGB |
|-------|----------------|--------|
| Transport | Kernel `hid_hw_raw_request()` | Userspace `ioctl(HIDIOCSFEATURE)` via hidraw |
| Report ID | `buf[0] = 0xCC` | `payload[0] = 0xCC` |
| Buffer size | 64 bytes | 64 bytes |

KeyRGB reuses the `ite8910.hidraw.HidrawFeatureTransport`, which uses `fcntl.ioctl(fd,
HIDIOCSFEATURE(len), payload)`. This is the standard Linux userspace HID feature report path
and produces the same USB control transfer as the kernel driver.

**Finding:** ✅ Correct.

---

### 4. RGB field order

Both implementations write red, green, blue at offsets 4, 5, 6 respectively.

**Finding:** ✅ Exact match.

### 5. Brightness model

| Aspect | tuxedo-drivers | KeyRGB |
|--------|----------------|--------|
| Per-channel brightness | 0–255 via `led_classdev` | N/A (uniform write) |
| Color/brightness coupling | Channel brightness IS the color | Color and brightness are separate |
| Turn-off method | Write (0, 0, 0) | `scale_color_for_brightness` returns (0,0,0) |

TUXEDO registers three separate `led_classdev` devices (`ite_8297:1`, `ite_8297:2`,
`ite_8297:3`) with `max_brightness = 0xff`. Each channel write triggers a full
`ite8297_write_color()` with the current R/G/B state.

KeyRGB takes a different policy approach: it keeps RGB color and UI brightness (0–50) as
separate concepts, then scales the RGB values proportionally before sending the single HID
report. Both approaches produce valid HID packets.

**Finding:** ✅ Policy difference, not a protocol bug. KeyRGB's approach fits its UI model
where color and brightness are independent user controls.

---

### 6. LED class device names (sysfs交叉引用)

TUXEDO registers `ite_8297:1` (red), `ite_8297:2` (green), `ite_8297:3` (blue) as sysfs LED
class devices. KeyRGB's `sysfs-leds` backend detects these names and handles them as a
channel triplet.

The KeyRGB `ite8297` HID backend bypasses sysfs entirely and writes directly via hidraw.
Both paths are valid and produce the same hardware state. The sysfs path is used when the
TUXEDO kernel module is loaded; the HID path is used as a userspace alternative.

**Finding:** ✅ Correct architecture. The two KeyRGB backends (`sysfs-leds` for kernel-driven,
`ite8297` for userspace HID) do not conflict because the registry picks one based on priority
and availability.

---

### 7. Capabilities

```python
BackendCapabilities(per_key=False, color=True, hardware_effects=False, palette=False)
```

This matches the TUXEDO driver, which only supports uniform color. No per-key, no effects,
no palette.

**Finding:** ✅ Correct.

---

### 8. Error handling and exception transparency

The backend reuses the standard `_open_matching_transport()` → `BACKEND_OPEN_RUNTIME_ERRORS`
pattern with `@quality-exception` annotation for permission/disconnected/busy/IO translation.
This is consistent with all other experimental ITE backends.

The `_send()` method checks `result == -1` rather than `result < 0` (as some sibling backends
do). This is a minor inconsistency but not a bug: the `HidrawFeatureTransport` returns
`len(payload)` (positive) on success or raises `OSError` on failure; it never returns -1
without raising.

**Finding:** ✅ Correct. The `== -1` check is defensively harmless.
