## 0.19.0 Reference: Other Useful Remaining Changes

Last updated: 2026-03-29

## Scope

This note captures the remaining `0.19.0` ideas that still look useful after the `0.18.5` tray/docs/assets backport, but do not belong in the memory/typing bucket.

Explicitly excluded here:

- the `0.19.0` `ite8910` backend implementation and policy promotion work
- changes already backported in `0.18.5`
- formatting-only diffs

## Highest-value remaining item

### Backend-owned engine runtime boundaries

This is the most important leftover reference change.

`0.18.5` made tray state and tray effect selection backend-aware, but `main` still keeps part of the runtime execution path coupled to the legacy global effects adapter.

`0.19.0` cleaned up that boundary in a way that is still worth revisiting:

- `src/core/effects/device.py`
  - acquires the keyboard device from an injected backend or from backend auto-selection, instead of always using the legacy global `ite_backend.get()` path
- `src/core/effects/engine_core.py`
  - stores the selected backend on the engine
  - adds `set_backend()` so runtime backend changes force a safe reacquire
  - adds `get_backend_effects()` and `get_backend_colors()` accessors so engine start logic does not read global hardware-effect tables
- `src/core/effects/engine_start.py`
  - validates effect names against software effects plus the selected backend's exposed hardware effects
  - treats `hw:` names as forced hardware selections
  - rejects legacy generic hardware names when the selected backend does not actually expose them
  - builds hardware payloads and color data from the injected backend rather than from global `hw_effects` / `hw_colors`
- `src/core/effects/ite_backend.py`
  - removed in `0.19.0` after the above cleanup

Why this is useful:

- it completes the tray/backend-authoritative direction already started in `0.18.5`
- it removes the remaining split-brain risk where tray/backend state and engine runtime state can diverge
- it narrows legacy compatibility behavior instead of keeping the generic hardware catalog alive as if it were authoritative runtime data

This is the main leftover code change worth carrying forward later.

## Useful supporting tests

If the engine-boundary cleanup is picked up, these `0.19.0` tests are worth bringing with it:

- `tests/core/test_effects_engine_stop_unit.py`
  - verifies backend-injected hardware effect dispatch
  - verifies generic legacy hardware names are rejected when the selected backend does not expose them
- `tests/core/test_effect_catalog_unit.py`
  - verifies config-name resolution and fallback behavior for stale legacy effect names

These tests are useful even if the implementation is redone differently on current `main`.

## Secondary doc and workflow follow-ups

### 1. Backend confirmation issue template

The new `.github/ISSUE_TEMPLATE/experimental-backend-confirmation.yml` is a useful maintainer workflow addition independent of the rejected `ite8910` runtime code.

Why it is useful:

- gathers consistent evidence when users report successful experimental backend results
- makes promotion requests less ad hoc
- creates a reusable structure for future backend validation, not just `ite8910`

### 2. README clarifications worth reusing selectively

There are a few useful README deltas in `0.19.0` that are not tied to the rejected backend work:

- document that hardware effect names are backend-specific rather than a universal fixed list
- document the `hw:` storage prefix for hardware/software name collisions such as `hw:spectrum_cycle`
- clarify that `--update-appimage` refreshes the broader desktop integration, not just the binary payload

These are documentation follow-ups, not release-critical code work.

## Changes I would not carry over as-is

- `src/tray/ui/icon_draw.py` from `0.19.0` as written, because it restores a direct `cairosvg` import path that is unsafe for optional-dependency packaging
- `src/core/resources/defaults.py` and `src/core/resources/layout.py` simplifications from `0.19.0`, because they hard-code the reference assumptions more aggressively and drop the ISO extra-key layout coverage

## Recommended order if this work is revisited

1. Revisit the engine/backend runtime boundary cleanup.
2. Bring over the matching tests for effect resolution and injected-backend dispatch.
3. Cherry-pick the maintainer/docs follow-ups independently of the runtime work.
