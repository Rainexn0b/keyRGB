# 0.19.1 Branch Notes

Last updated: 2026-04-01

## Overview

`0.19.1` is a focused post-`0.19.0` quality pass. It contains no new user-visible features; every change is a bug fix, a behavioral regression correction, or a build-quality improvement discovered during a systematic review of the `0.19.0` code immediately after release.

All fixes target the software-effects and reactive-typing subsystems that were heavily reworked in `0.19.0`.

---

## Changes

### 1. Fix per-key software effects randomly bouncing between colors on `ite8291r3`

**File:** `src/core/effects/software/base.py`

**Root cause:** The `ite8291r3` backend silently reverts the controller to its saved hardware effect whenever user mode is not explicitly maintained. The `0.19.0` software-render loop called `enable_user_mode()` once at startup but did not reassert it before each `set_key_colors` call. Between frames the controller would fall back to its saved state, causing per-key color data to be overwritten by the hardware effect — visible as rapid bouncing or flickering between the software colors and the hardware saved state.

**Fix:** `SoftwareRenderBase.render()` now calls `kb.enable_user_mode()` immediately before every `set_key_colors` call. The ITE protocol requires this to be idempotent; the cost is one extra USB HID command per frame, which is not measurable vs the full frame write.

**Affected effects:** All per-key software effects on `ite8291r3`: rainbow swirl, rainbow wave, breathe, twinkle, strobe, color chase, spectrum cycle, color cycle, and any future additions.

---

### 2. Fix choppy high-speed appearance on hue-cycling effects

**File:** `src/core/effects/software/_effects_basic.py`

**Root cause:** Four hue-cycling effects (`rainbow_swirl`, `rainbow_wave`, `spectrum_cycle`, `color_cycle`) were updated in `0.19.0` to advance hue by `elapsed_wall_clock_time × pace`. At high speed the pace multiplier reaches ×10. USB write latency is variable (typically 4–16 ms) and is included in the measured elapsed time. At ×10 pace, a single frame's USB write variance could produce hue jumps of 40–160 degrees — visible as severe color stepping at max speed.

**Fix:** Hue advances by a constant `BASE_STEP × pace` per frame regardless of elapsed time. The rendering loop already calls `stop_event.wait(nominal_dt)` for pacing; this makes the step size independent of USB timing jitter.

---

### 3. Eliminate dark-dip flicker on software → software effect switches

**Files:** `src/core/effects/engine_start.py`

**Root cause:** `engine.start_effect()` unconditionally ran a fade-in animation when `fade_in=True` was passed. Calling code in `start_current_effect` used `fade_in=True` for all effect starts. When switching from one software effect to another, the engine's `stop()` → `start()` cycle briefly darkened the keyboard during the fade-in window even though the keyboard was already in user mode and there was no perceptual reason to dim.

**Fix:** `start_effect()` captures `prev_effect_was_sw` before calling `stop()`. A new `from_sw_effect` parameter is threaded through to `_start_sw_effect`. When `from_sw_effect=True`, all fade-in logic is bypassed and the first frame of the new effect is rendered immediately. The keyboard never leaves user mode between effects and the transition is seamless.

**Not affected:** Hardware → software and software → hardware transitions still perform the fade-in.

---

### 4. Per-effect speed persistence for software effects

**Files:** `src/core/config/config.py`, `src/core/config/defaults.py`, `src/tray/controllers/lighting_controller.py`

**What was missing:** In `0.19.0` the speed setting was a single global value. Switching effects reset the visible speed to whatever the global happened to be. There was no way to keep rainbow wave at speed 8 and spectrum cycle at speed 2 simultaneously.

**Implementation:**
- `DEFAULTS` gains `"effect_speeds": {}` — a per-effect dict, persisted with the rest of the config.
- `Config.get_effect_speed(name)` returns the per-effect override if set, otherwise the global `config.speed`.
- `Config.set_effect_speed(name, speed)` writes the per-effect override.
- `on_speed_clicked` in `lighting_controller.py` writes both the global speed and the per-effect override, then updates `engine.speed` in-place for loop effects (software and reactive) without restarting the loop.
- `start_current_effect` passes `tray.config.get_effect_speed(effect)` to `engine.start_effect()` so the engine starts at the correct per-effect speed.
- After `engine.start_effect()`, `config.speed` is synced to the effective per-effect speed so the tray speed-menu checkmark reflects the running speed immediately.

**Upgrade path:** Existing configs without `effect_speeds` behave identically to before — `get_effect_speed` falls back to the global speed.

### 5. Fix reactive typing speed changes having no visible effect

**File:** `src/core/effects/reactive/effects.py`

**Root cause:** Both `_reactive_fade_loop` and `run_reactive_ripple` called `pace(engine)` once before entering their while loop. The `on_speed_clicked` path for loop effects updates `engine.speed` in-place to avoid restarting the loop. Because `p` was captured as a float at startup, loop-local values like pulse TTL (`0.48 / p`, `0.65 / p`) and synthetic press spawn interval (`0.45 / p`) never changed for the lifetime of the effect.

**Fix:** `p = pace(engine)` is moved inside both while loops so it is re-evaluated every frame. `press.spawn_interval_s` is updated each frame to match. Pulse TTL is already computed per-keypress from the current `p`, so no further change is needed there.

**Interaction with per-effect speed:** This fix is necessary for the per-effect speed feature (change 4 above) to be observable in reactive effects; without it, setting a new speed while reactive is running would have no effect until the effect was restarted.

---

### 6. Fix speed menu checkmark showing wrong value after effect switch

**File:** `src/tray/controllers/lighting_controller.py`

**Root cause:** The tray speed menu uses `tray.config.speed` (global) to determine which speed item gets a checkmark. After switching to an effect whose per-effect speed differs from the previous global speed, `config.speed` still held the old value. The checkmark pointed at the old speed even though the engine was running at the per-effect speed.

**Fix:** After `engine.start_effect()` completes, `start_current_effect` reads `get_effect_speed(effect)` and writes it back to `config.speed` if they differ. The config poller detects the speed change, triggers one seamless SW→SW restart (the dark-dip fix makes this invisible), and settles — no further restarts occur because the new `config.speed` matches the effective speed.

---

### 7. Fix mypy errors in `config_polling_internal/core.py`

**File:** `src/tray/pollers/config_polling_internal/core.py`

**Issue:** `tuple(color)` and `tuple(reactive_color)` returned `tuple[int, ...]` but the `ConfigApplyState` dataclass fields are typed `tuple[int, int, int]`. These were pre-existing errors that caused `buildpython`'s full profile (which includes mypy) to fail.

**Fix:** Added `from typing import cast` and wrapped both tuple constructions with `cast("tuple[int, int, int]", tuple(...))`. Runtime behavior is unchanged; the cast is a type-checker annotation only.

---

### 8. Fix mypy errors in `buildpython/core/summary.py`

**File:** `buildpython/core/summary.py`

**Issues:**
- Variable `category` used as a loop variable in the regressions section at line 229, shadowing the outer loop variable of the same name — mypy `[no-redef]` error.
- `parts: list[str] = []` defined at line 335 and redefined at line 387 in the same function — mypy `[no-redef]` error.

**Fix:**
- Renamed the inner loop variable from `category` to `reg_category` in the regressions block.
- Renamed the second `parts` definition to `et_parts` and updated all references in the `exception_transparency` block.

### 9. Debt baseline updates

**File:** `buildpython/config/debt_baselines.json`

Two baseline values updated to reflect intentional patterns that should not regress further:

- `forbidden_getattr`: raised from 51 → 54. The three new counts are in `src/gui/windows/reactive_color.py` — `getattr(self, "_last_drag_commit_ts", ...)` and friends. These are intentional defensive reads validated by `test_on_color_change_tolerates_missing_drag_state`, which constructs a `ReactiveColorGUI` via `__new__` without calling `__init__`. They cannot be replaced with direct attribute access without breaking the resilience contract.
- `logged_broad_except` path budget for `src/tray/pollers/config_polling_internal/core.py`: raised from 3 → 5. The five logged broad-except handlers in the config polling apply loop are all throttled (max one log per 60 s) and correctly typed as `Exception as exc` with explicit log calls. They are correct design for a polling loop that must not crash the tray on transient hardware errors.
